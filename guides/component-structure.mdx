Every React component in Pragma follows the same seven-file structure. Learning one component teaches you how to navigate any component in the monorepo.

## Standard folder layout

The folder name matches the component name exactly. All files related to the component live together inside it.

```
Button/
├── Button.tsx           # Component implementation
├── types.ts             # TypeScript interface
├── styles.css           # Component styles
├── index.ts             # Barrel export
├── Button.stories.tsx   # Storybook stories
├── Button.tests.tsx     # Unit tests
└── Button.ssr.tests.tsx # Server-side rendering tests
```

## Subcomponent structure

Components with secondary subcomponents add a `common/` directory. Subcomponents inside `common/` follow the identical seven-file layout and exist to support the parent rather than stand alone.

```
Tooltip/
├── Tooltip.tsx
├── types.ts
├── styles.css
├── index.ts
├── Tooltip.stories.tsx
├── Tooltip.tests.tsx
├── withTooltip.tsx           # Higher-order component variant
├── withTooltip.stories.tsx
└── common/
    ├── index.ts
    └── TooltipArea/
        ├── TooltipArea.tsx
        ├── types.ts
        ├── styles.css
        ├── index.ts
        ├── TooltipArea.stories.tsx
        └── TooltipArea.tests.tsx
```

## Creating a new component

<Steps>
  <Step title="Run the generator">
    From inside a component package directory, use the component generator to scaffold all required files:

    ```bash
    yo @canonical/ds:component src/ui/MyComponent
    ```

    Include optional flags as needed:

    ```bash
    # Include styles and stories
    yo @canonical/ds:component src/ui/MyComponent --withStyles --withStories

    # Exclude SSR tests (e.g. for components that are client-only by design)
    yo @canonical/ds:component src/ui/MyComponent --withoutSsrTests
    ```

    See [Generators](/tooling/generators) for installation and all available flags.
  </Step>

  <Step title="Implement the component">
    Open `MyComponent.tsx` and implement the component following the patterns described below. The generated scaffold already imports `./types.js` and `./styles.css`.
  </Step>

  <Step title="Define the types">
    Edit `types.ts` to extend the appropriate HTML attributes interface and declare component-specific props.
  </Step>

  <Step title="Add styles">
    Edit `styles.css` with component-scoped rules that reference design tokens via CSS custom properties.
  </Step>

  <Step title="Write tests">
    Add behaviour tests in `MyComponent.tests.tsx` and SSR tests in `MyComponent.ssr.tests.tsx`.
  </Step>

  <Step title="Update the barrel export">
    Edit `index.ts` to add named exports for the component and its public types.
  </Step>
</Steps>

## The implementation file

The component file imports its types and styles, then exports a default function component.

```typescript
import type Props from "./types.js";
import "./styles.css";

const Button = ({
  id,
  className,
  children,
  style,
  appearance,
  ...props
}: Props): React.ReactElement => {
  return (
    <button
      id={id}
      className={["ds", "button", appearance, className]
        .filter(Boolean)
        .join(" ")}
      style={style}
      aria-label={props["aria-label"] || children?.toString()}
      {...props}
    >
      {children}
    </button>
  );
};

export default Button;
```

### Key conventions

**`.js` extension on imports** — TypeScript compilation produces `.js` files; using `.js` in source imports ensures the compiled output resolves correctly.

**CSS import without a specifier** — `import "./styles.css"` is a side-effect import that loads styles. There is no exported value.

**Default export** — The barrel file re-exports the default as a named export, so consumers can write `import { Button } from "@canonical/react-ds-global"`.

### className construction pattern

All components build `className` using an array that is filtered for falsiness and joined with spaces:

```typescript
className={["ds", "button", appearance, className]
  .filter(Boolean)
  .join(" ")}
```

The array contains, in order: the `ds` namespace, the component class, any modifier props (e.g. `appearance`), and any consumer-provided `className`. `filter(Boolean)` removes `undefined` when a modifier prop is omitted. When `appearance="positive"` is passed, the rendered element receives `class="ds button positive"`, which activates the intent colours defined in the style system.

## The types file

Props interfaces extend the matching HTML attributes type so all native attributes pass through without explicit declaration.

```typescript
import type { ModifierFamily } from "@canonical/ds-types";
import type { ButtonHTMLAttributes, ReactNode } from "react";

export interface BaseProps {
  id?: string;
  className?: string;
  children: ReactNode;
  appearance?: ModifierFamily<"severity"> | "base" | "link";
}

type Props = BaseProps & ButtonHTMLAttributes<HTMLButtonElement>;

export default Props;
```

`BaseProps` declares the component-specific props that appear in Storybook controls and IDE autocomplete. The intersection with `ButtonHTMLAttributes<HTMLButtonElement>` adds all standard button attributes automatically.

`ModifierFamily<"severity">` comes from [`@canonical/ds-types`](/reference/ds-types) and resolves to the union `"neutral" | "positive" | "negative" | "caution" | "information"`. It is derived from the design system ontology, so changes to modifier families propagate to TypeScript as compile errors.

## The styles file

Component styles use CSS custom properties that reference design tokens. The `.ds.button` compound selector combines the namespace and component class to prevent accidental conflicts.

```css
.ds.button {
  display: inline-block;
  cursor: pointer;

  color: var(--modifier-color-text, var(--button-color-text));
  background-color: var(--modifier-color, var(--button-color-background));
  border-color: var(--modifier-color-border, var(--button-color-border));
  border-style: solid;
  border-width: var(--button-border-width);
  font-size: var(--button-font-size);
  padding-block: var(--button-padding-vertical);
  padding-inline: var(--button-padding-horizontal);

  &:hover {
    background-color: var(
      --modifier-color-hover,
      var(--button-color-background-hover)
    );
  }
}
```

The nested fallback pattern `var(--modifier-color, var(--button-color-background))` enables intent inheritance: when the button is inside a container with the `positive` class, `--modifier-color` resolves to the intent colour. Outside any intent context the button falls back to `--button-color-background`.

Consumers can override individual tokens via inline styles:

```jsx
<Button style={{ "--button-color-background": "lightblue" }}>
  Custom
</Button>
```

## The barrel export

The `index.ts` file explicitly lists every public export. No `export *` statements are used.

```typescript
export { default as Button } from "./Button.js";
export type {
  BaseProps as ButtonBaseProps,
  default as ButtonProps,
} from "./types.js";
```

The default export from `Button.tsx` becomes the named export `Button`. Type exports use `export type` so they are erased at compile time. Types are renamed on export (e.g. `BaseProps as ButtonBaseProps`) to include the component name and prevent collisions when multiple components define similarly named base props.

## Stories

Storybook stories document usage and provide a visual testing surface for Chromatic.

```typescript
import type { Meta, StoryObj } from "@storybook/react-vite";
import { fn } from "storybook/test";

import Component from "./Button.js";

const meta = {
  title: "Button",
  component: Component,
  tags: ["autodocs"],
  args: { onClick: fn() },
} satisfies Meta<typeof Component>;

export default meta;
type Story = StoryObj<typeof meta>;

export const Default: Story = {
  args: {
    children: "Contact us",
  },
};

export const Positive: Story = {
  args: {
    children: "Confirm",
    appearance: "positive",
  },
};
```

`tags: ["autodocs"]` instructs Storybook to generate API reference documentation from JSDoc comments and TypeScript types. Each exported story represents a distinct state or variant; descriptive names like `Positive` and `Negative` help reviewers understand each Chromatic screenshot.

## Unit tests

Unit tests verify behaviour using Testing Library's user-centric queries.

```typescript
import { render, screen } from "@testing-library/react";
import { describe, expect, it } from "vitest";
import Component from "./Button.js";

describe("Button component", () => {
  it("renders", () => {
    render(<Component>Hello world!</Component>);
    expect(screen.getByText("Hello world!")).toBeInTheDocument();
  });

  it("applies className", () => {
    render(<Component className="test-class">Hello world!</Component>);
    expect(screen.getByText("Hello world!")).toHaveClass("test-class");
  });
});
```

Tests focus on observable behaviour rather than implementation details. Queries like `getByText` and `getByRole` interact with components the way users do. If an element cannot be found by its accessible role or text, that often indicates an accessibility problem.

## SSR tests

SSR tests verify that components render correctly without browser APIs using `renderToString` from `react-dom/server`.

```typescript
import { renderToString } from "react-dom/server";
import { describe, expect, it } from "vitest";
import Component from "./Button.js";

describe("Button SSR", () => {
  it("doesn't throw", () => {
    expect(() => {
      renderToString(<Component>Hello world!</Component>);
    }).not.toThrow();
  });

  it("renders", () => {
    const html = renderToString(<Component>Hello world!</Component>);
    expect(html).toMatch(/<button[^>]*>Hello world!<\/button>/);
  });

  it("applies className", () => {
    const html = renderToString(
      <Component className="test-class">Hello world!</Component>,
    );
    expect(html).toContain('class="ds button test-class"');
  });
});
```

The "doesn't throw" test catches the most common SSR failure: direct access to `window`, `document.querySelector`, or other browser globals executed during render. Components that require browser APIs must guard those accesses with environment checks or move them into effects that run only on the client.

HTML string assertions catch hydration mismatches before they cause warnings in production.

## Why this structure

Three principles drive the layout:

- **Co-location** — everything related to a component lives in one folder. Modifying, testing, and deleting a component all operate on a single directory.
- **Predictable naming** — the component file matches the component name. No memorisation is required to find `Button.tests.tsx` or `Button.stories.tsx`.
- **Explicit exports** — `index.ts` lists exactly what the component exposes. The public API is visible at a glance without `export *` surprises.
